# News QA — 통합 정합성 검증

뉴스 모니터링 시스템에서 가장 위험한 결함은 **경계면 불일치**다. 각 모듈이 개별로는 정상이어도 연결 지점에서 계약이 어긋나면 런타임에서 깨진다. 이 스킬은 "존재 확인"이 아니라 "교차 비교"를 수행한다.

## 핵심 원칙: 양쪽 동시 읽기

한쪽만 읽으면 경계면 버그를 못 잡는다. 반드시 생산자와 소비자를 같이 열어 비교한다.

| 검증 대상 | 왼쪽(생산자) | 오른쪽(소비자) | 흔한 버그 |
|----------|-------------|---------------|----------|
| NewsItem shape | collector 정규화 출력 | 스키마, filter 입력 | 필드명 오타, 누락 |
| 점수·근거 | filter 출력(importance_*) | dashboard 알림 카드 | 빈 reason, None score 표시 |
| API 응답 | FastAPI route 반환 | 대시보드 JS fetch 파싱 | 래핑 불일치(`{items}` vs 배열) |
| 카테고리 enum | 스키마 상수 | 각 모듈 분기 코드 | 문자열 오타로 분기 누락 |
| 임계값 형식 | config.yaml | filter 읽기 + 설정 UI | 키 이름 불일치 |

## 점진적 검증 (Incremental QA)

빌드 후 1회가 아니라 **각 모듈 완성 직후** 검증한다. 초기 경계면 불일치가 후속 모듈로 전파되는 것을 막는다.

1. collector 완성 → 정규화 출력 NewsItem이 스키마와 일치하는지 (필드명·타입·id 규칙)
2. filter 완성 → 입력(collector 출력) 소비가 맞는지 + 출력 필드(importance_*, send_recommended)가 스키마대로인지
3. dashboard 완성 → API 응답 shape ↔ 프론트 fetch 파싱 교차 검증

## 통합 정합성 체크리스트

### 데이터 스키마 정합성
- [ ] collector가 채우는 필드명이 스키마와 정확히 일치 (snake_case)
- [ ] id가 `sha256(url)[:16]` 규칙대로 생성 (24h 재발송 차단의 기준)
- [ ] category가 3개 enum 값만 사용, 모든 분기 코드가 동일 상수 참조
- [ ] published_at/collected_at이 ISO 8601 형식으로 일관

### 필터 파이프라인 정합성
- [ ] 4단계 순서대로 연결 (키워드→점수→임계값→중복제거)
- [ ] 모든 발송 후보의 importance_reason이 비어있지 않음 (설명 가능성)
- [ ] 임계값을 config에서 읽음 (하드코딩 없음), 키 이름이 config와 일치
- [ ] global_finance_ai 월 1건 캡 로직 존재

### API ↔ 대시보드 정합성
- [ ] 리스트 API가 `{items:[...], total}`로 래핑되고 프론트가 `.items`를 꺼냄
- [ ] API 응답 필드명 = 스키마 = 프론트 파싱 (3자 일치)
- [ ] 모든 API 엔드포인트에 대응하는 프론트 호출이 존재하고 실제 호출됨
- [ ] 빈 데이터에서 대시보드가 빈 상태를 정상 렌더 (배열 보장)

### 발송 로직 정합성
- [ ] 이벤트 기반(정기 스케줄 없음), 임계값 초과 시에만 발송
- [ ] 24h 동일 news_id 재발송 차단이 send_history로 동작
- [ ] 묶음 발송 batch_id가 send_history에 기록

## 발견 시 행동

- 발견 즉시 해당 에이전트에게 **파일:라인 + 수정 방법**으로 수정 요청
- **경계면 이슈는 양쪽 에이전트 모두**에게 알린다 (한쪽만 고치면 또 어긋남)
- 검증 불가(미완성)는 "미검증"으로 표시, 임의 판단하지 않음
