# News Source Research — 소스 실제 검증·선정

뉴스 소스를 "좋아 보인다"가 아니라 **실제로 fetch·파싱한 측정값**으로 채택/탈락 판정하는 스킬. collector-dev는 주어진 소스를 구현만 하므로, 어떤 소스를 쓸지는 이 스킬로 결정한다.

## 산출물
- `_workspace/00b_source_research.md` — 채택/탈락 소스 목록 + 측정 근거 + 신뢰 출처 화이트리스트
- collector-dev의 `references/sources.md` 입력으로 직결되고, 빌드 모드에서 architect가 참조한다.

## 검증 절차
1. **후보 수집** — WebSearch로 카테고리(global_ai / global_finance_ai)별 RSS/API/뉴스 페이지 후보를 모은다.
2. **robots.txt·약관 검토** — 크롤링 검증 전 robots.txt와 이용약관을 확인한다. 차단/금지 소스는 강행하지 않고 탈락 사유에 명시한다.
3. **실제 fetch·파싱**
   - RSS 후보 → `feedparser`로 파싱, 항목 수·최신 날짜 확인
   - HTML/JS 렌더링 페이지 → `Scrapling`(StealthyFetcher)로 fetch 후 구조 확인
   - 내부 JSON API → `httpx`로 응답 형태 확인
4. **측정·점수화** — 아래 지표를 소스별로 기록한다.
5. **판정** — 채택/탈락 + 근거를 산출하고 채택분을 화이트리스트로 정리한다.

## 측정 지표
| 지표 | 측정 방법 | 탈락 기준 |
|------|----------|----------|
| 파싱 성공 | 실제 fetch 후 항목 추출 | 죽은/형식 깨진 피드 |
| 발행 빈도 | 일/주당 항목 수 | 거의 갱신 없음 |
| 최신성 | 최근 항목 발행일 | 수개월 정체 |
| 카테고리 적합도 | 관련 기사 비중 | 무관 기사 다수 |
| 추정 중복률 | 다른 채택 소스와 동일 기사 비중 | 중복 과다 → 탈락/보조 강등 |

## 점수 기준
- 5개 지표를 종합하여 **채택 / 보조(중복 높지만 보완용) / 탈락** 3단계로 판정한다.
- 채택 우선순위: 파싱 성공 + 최신성 + 카테고리 적합도가 모두 양호한 소스.
- 동률이면 발행 빈도가 적당하고(과다 아님) 중복률이 낮은 소스를 우선한다.

## 채택 소스 기재 형식
채택 소스는 collector-dev가 바로 구현할 수 있도록 다음을 명시한다: URL, 수집 방식(feedparser/Scrapling/httpx), 카테고리, 비고(특이 파싱 사항).

## 작업 원칙
- 추측 금지 — 모든 판정은 실제 fetch 결과 수치에 근거한다.
- 차단 소스는 강행하지 않는다 (법적·기술적 제약 명시).
- 미결 소스(예: NH농협)는 확인 불가 시 "TODO/확장 포인트"로 남기고 채택 가능한 소스로 진행한다.

## 후속/재실행
