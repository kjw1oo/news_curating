# News Threshold Tuning — 피드백 기반 임계값 튜닝 제안

운영 중 누적된 `feedback`(FP/FN)과 `send_history`를 분석하여 카테고리별 임계값·평가 프롬프트 조정안을 만드는 스킬. 핵심 안전 원칙은 **"제안만 한다 — 자동 반영하지 않는다"**.

## 산출물
- `_workspace/06_tuning_proposal.md` — 카테고리별 현재값 → 제안값 + 근거 + 예상 효과(발송 빈도 변화 추정)
- 운영자 승인 시에만 `config.yaml` 임계값/프롬프트 패치 + filter 재실행.

## 분석 방법
1. **데이터 수집** — SQLite의 `feedback`(오발송 FP / 누락 FN)과 `send_history`를 읽는다.
2. **카테고리별 집계** — global_ai / global_finance_ai / domestic_finance_ai 별로 발송 건수, FP 건수, FN 건수를 집계한다.
3. **경향 진단** — 과발송(FP 다수) vs 누락(FN 다수) 방향을 판단하고, 어떤 *유형*의 기사가 문제인지 패턴을 찾는다.
4. **조정안 도출** — 과발송 경향이면 임계값 상향 또는 평가 프롬프트 강화, 누락 경향이면 하향 또는 프롬프트 완화를 제안한다.
5. **예상 효과 추정** — 제안값 적용 시 발송 빈도가 어떻게 변할지 추정한다.

## 제안 형식
| 카테고리 | 현재값 | 제안값 | 근거(FP/FN 패턴) | 예상 효과 |
|----------|--------|--------|------------------|-----------|
| global_ai | … | … | … | … |
- 임계값 외 평가 프롬프트 조정 제안은 "현재 문구 → 제안 문구 → 사유"로 별도 기재한다.

## 오버피팅 방지 (불변 원칙)
- 소수 피드백 1~2건에 맞춘 좁은 규칙을 만들지 않는다.
- "이 기사 1건이 잘못 발송됨"이 아니라 **"이런 유형이 과발송/누락되는 경향"** 수준으로 원리를 일반화한다.
- 표본이 적으면 "표본 부족 — 더 많은 피드백 누적 후 재검토"로 명시하고 무리한 조정을 제안하지 않는다.
- 변경은 최소·가역적으로 (한 번에 임계값 대폭 변경 금지).

## 안전성 (불변 원칙)
- **자동 반영 금지.** config는 운영자 승인 후에만 패치한다 (요구사항 6.2 "운영자 조정 가능"과 정합).
- 승인 전에는 어떤 설정 파일도 수정하지 않는다.

## 후속/재실행
- `_workspace/06_tuning_proposal.md`가 이미 존재하면 Read하여 이전 제안·적용 여부를 파악하고, 신규 피드백 누적분만 반영하여 재제안한다.
